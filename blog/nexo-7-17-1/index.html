NEXO 7.17.1 — handle Claude CLI 2.1+ direct-JSON response shape

Published 2026-05-10. Patch release over v7.17.0.

Silent daily failure

The morning-agent cron has been failing every tick with Morning agent returned invalid JSON output — 178 log lines on 2026-05-10 alone. The peculiar part: automation_runs showed 75 status=ok rows for the same runs, each carrying a perfectly formed subject+body in metadata.raw. Every fail had a matching successful telemetry row at the same wall-clock second. The agent had answered correctly. The caller had not received the answer.

Root cause

Claude CLI 2.1+ with bare_mode=True + output_format="json" + a prompt asking for raw JSON drops the classic {"result": "...", "usage": {...}} wrapper. The entire stdout becomes the agent's JSON answer ({"subject":..., "body":...}). _extract_claude_telemetry did payload.get("result", ""), which now returned an empty string. The empty final_stdout reached morning-agent's downstream JSON parser, which raised "Morning agent returned invalid JSON output". Telemetry persistence already happened earlier, which is why the DB rows looked healthy while the caller died.

Fix

When payload has no "result" key, treat the entire payload as the agent's answer. Surface it as final_stdout, serialised as JSON when output_format="json". Telemetry of usage/cost is unavailable in this shape; we surface zeros rather than invent figures. Classic-wrapper callers are untouched — pinned by the existing _claude_json_result-based tests.

Validation

New tests pin: the classic wrapper passes result through unchanged; the direct-JSON shape surfaces the full payload as final_stdout; the morning-agent's _extract_json_payload can consume the result without raising. Three regressions impossible to come back.
