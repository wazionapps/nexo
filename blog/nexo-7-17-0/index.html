NEXO 7.17.0 — pre-emptive runner guard becomes advisory only

Published 2026-05-10. Minor release over v7.16.3.

The bigger picture behind 7.16.1 → 7.16.3

Three patches in the same day chased the same shape of bug: the headless runner's pre-emptive guard was treating heuristic path matches in the prompt as hard blockers. v7.16.2 closed the subprocess-invocation pattern. v7.16.3 closed the runtime-core duplication. v7.17.0 closes the family.

What changed

_run_headless_runner_guard is now advisory: it surfaces relevant learnings and schemas to the agent up front and logs to guard_checks for observability, but never returns blocked=True. Even an unhandled exception inside handle_guard_check no longer aborts the run. The authoritative gate stays where it should be: the PreToolUse hook (hook_guardrails._collect_runtime_core_write_blocks and the file-conditioned learning blocks alongside it). That hook fires only when the agent actually attempts a Write/Edit, with severity error.

Why this is the right place to enforce

The pre-emptive guard scans prompt text. It cannot tell whether a path will actually be written, read, executed, or just mentioned in passing. Treating that heuristic as enforcement caused a year's worth of operational pain — cron jobs aborting once an hour, Deep Sleep synth aborting nightly, email-monitor demoting every forwarded message to needs_interactive, postmortem-consolidation aborting at 23:30. PreToolUse runs at write time with full intent context. The two layers no longer compete.

Defence in depth from 7.16.4

This release also rolls in the directory-path hardening that was scheduled for 7.16.4 but consolidated here: handle_guard_check skips paths that resolve to a directory on disk (Path.is_dir()), the except clause covers OSError, and the path extractor strips trailing slashes. So even before the advisory contract reaches every caller, a /tmp/ mention can no longer crash the schema scan.
