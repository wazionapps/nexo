NEXO 7.16.2 — runner guard no longer blocks email-monitor sessions

Published 2026-05-10. Patch release over v7.16.0.

What was breaking

Every forwarded email landed at status='needs_interactive' after three failed attempts, regardless of subject or sender. Operators received the manual-attention notification but never saw a single reply leave the system. Sessions aborted with exit 2 on the runtime-core blocking rule before the agent could draft a response.

Root cause

The pre-emptive runner guard scanned the prompt for absolute paths and treated every match as an edit target. The email-monitor template instructs the agent with python3 /Users/.../core/scripts/nexo-send-reply.py --to ..., so the path of the reply tool itself entered the guard's file list, the runtime-core rule fired, and the session aborted. The script was being invoked, not edited — but the extractor could not tell.

Fix

_extract_runner_guard_paths now recognises subprocess invocations: paths immediately following a known interpreter (python, python3, node, bash, sh, npx, pnpm, yarn, uv, pipx, env, pwsh, powershell, …) are excluded from the guard list. Paths in genuine edit instructions and other CLI arguments (body files, quote files, etc.) still go through the guard.

Validation

python3 -m pytest tests/test_runner_guard_path_extraction.py -q

Seven tests pin python/node/npx exclusions, edit-instruction capture, mixed exec+edit prompts, and a direct reproduction of the email-monitor template scenario.
