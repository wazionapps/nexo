NEXO 7.36.0 - Local index disk reclaim: no more unbounded growth

v7.36.0 closes the one place NEXO Brain leaked disk for every user: the local file/code index. A disk audit confirmed the backup subsystem is already well bounded — but the index (local-context.db) grew monotonically into tens of GB and never gave the space back.

Why it grew forever. The index database used auto_vacuum=NONE and ran no VACUUM anywhere, so when files were re-indexed or removed the freed pages only moved to an internal free-list — the file on disk never shrank. Embeddings were stored as JSON text (several times larger than necessary). Purging or clearing the index didn't return disk. And the daily self-audit only printed a warning at 25/60 GB while taking no action — a 268 GB runaway once went unnoticed.

What changed. New databases are created with auto_vacuum=INCREMENTAL, and an existing index is converted once via a disk-guarded full VACUUM, so deletes become reclaimable space. Embeddings are now stored as compact little-endian float32 BLOBs (~4-6x smaller than the JSON text). The change is fully back-compatible: a new column is added next to the legacy one, writes fill both, reads prefer the BLOB and fall back to JSON, a resumable background job converts old rows one batch at a time, and kill switches revert to JSON instantly if ever needed.

Reclaim, not just slower growth. Clearing the index now compacts and returns the space to the OS; purging an asset reclaims its pages incrementally; and the daily self-audit now acts at its size cap (checkpoint + incremental vacuum, counting the WAL sidecars) instead of only warning. An established index reclaims roughly 10-20 GB immediately and grows several-fold slower thereafter.
