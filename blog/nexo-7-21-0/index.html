NEXO 7.21.0 — resident Runtime Service for MCP stability

Published 2026-05-17. Minor release over v7.20.25.

Why this release exists

Running several MCP clients at the same time can create several full Brain processes. Each process initializes the runtime and may contend for the same local SQLite databases. v7.21.0 keeps the public MCP entrypoint compatible, but moves real execution behind one resident local Runtime Service.

What changed

  • Thin MCP adapter. Existing stdio MCP clients keep the same tool names and schemas, while the adapter proxies requests to a resident service over loopback HTTP.
  • Single runtime authority. The resident service owns Brain initialization and tool execution, reducing duplicate processes and SQLite contention across Claude Code, Codex, Claude Desktop and NEXO Desktop.
  • Safe update cutover. Runtime Service state records PID, URL, version, fingerprint and server path. New adapters reject stale services after an update and start the current runtime.
  • Cross-platform startup lock. macOS and Windows clients cannot race into launching multiple resident services at once.
  • Better diagnostics. MCP status now exposes Runtime Service readiness, stale state, fingerprint and log/state paths.
  • Local Memory and models hardened. User document folders stay prioritized, and bundled local LLM files are checked by size and SHA256 before Brain writes model locks.
