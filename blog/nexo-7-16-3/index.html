NEXO 7.16.3 — runner pre-emptive guard stops duplicating runtime-core protection

Published 2026-05-10. Patch release over v7.16.2.

Why this followup landed so quickly

v7.16.2 fixed the immediate symptom — the email-monitor template invokes python3 ~/.nexo/core/scripts/nexo-send-reply.py, the path was being captured as an edit target, and every session aborted. The fix taught the runner's path extractor about subprocess invocations. But that was a heuristic over text: a literal mention of a core path without an interpreter would still trip the same rule.

Architectural fix

handle_guard_check now accepts enforce_runtime_core_block="false". The headless runner pre-emptive guard passes that flag because the PreToolUse hook already blocks actual writes on ~/.nexo/core/ paths with severity error (hook_guardrails._collect_runtime_core_write_blocks). Two layers were enforcing the same protection; only one of them needed the runtime ground truth, and the other was relying on text heuristics. The duplicated layer steps back. The agent still cannot edit core — the PreToolUse hook stops every Write/Edit that touches a protected path. The pre-emptive guard now focuses on file-conditioned learnings only.

Default callers keep historical behaviour

Direct nexo_guard_check calls from agents still surface the runtime-core blocking rule. Only the headless runner opts out, with the safety tradeoff documented in the function docstring. Existing automations and red-team coverage that rely on the runtime-core block remain unchanged.
