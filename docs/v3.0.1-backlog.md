# NEXO v3.0.1 Backlog

Updated: 2026-04-06
Scope: post-release polish only, not a new roadmap reset

## Why this exists

`v3.0.0` closed the strategic roadmap and shipped successfully.
This file captures the next small, non-blocking improvements so they do not get mixed back into release-critical work.

## Priority order

### 1. Runtime update should preserve declared personal schedules automatically

Problem:
- after syncing `~/claude` from source, declared personal schedules may need an explicit `nexo scripts ensure-schedules`
- that is acceptable for `v3.0.0` closure but should not remain manual forever

Target:
- `nexo update` finishes with declared personal schedules present in LaunchAgents/systemd and in `personal_script_schedules`
- doctor should not require a second manual repair after a normal runtime sync

### 2. Merge and ship PR #18 if final maintainer review stays clean

PR:
- `#18 doctor: validate all critical config files at boot tier`

Why:
- broken `crons/manifest.json` or `optionals.json` can currently hide reliability loss until much later than boot
- the PR fixes a real gap and passed local review slices during post-release review

### 3. Remove remaining deprecated `datetime.utcnow()` usage

Problem:
- tests are green, but warnings remain in `cognitive/_search.py` and `cognitive/_memory.py`

Target:
- replace deprecated naive UTC calls with timezone-aware UTC timestamps
- keep outputs and DB behavior stable

### 4. Migrate FastAPI startup hook to lifespan

Problem:
- dashboard still emits deprecation warnings for `@app.on_event("startup")`

Target:
- move dashboard startup initialization to lifespan handlers
- keep behavior identical

### 5. Tighten release-readiness logging around runtime doctor subprocess output

Problem:
- during `v3.0.0` closure, `verify_release_readiness.py` sometimes printed a stale-looking `runtime doctor: degraded (1 issues)` line even when the live runtime was healthy after follow-up verification

Target:
- make release-readiness output unambiguous
- ensure reported doctor status matches the actual active runtime check that gates release decisions

## Not for v3.0.1 unless they regress

- no new large roadmap expansion
- no dashboard redesign
- no new memory/execution subsystem
- no public feature inflation without measured need
