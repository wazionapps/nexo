# NEXO Brain Cognitive Cortex — v0.1 Specification

> Designed: 2026-03-27 | Source: 3-way debate (Claude Opus 4.6 + GPT-5.4 + Gemini 3.1 Pro)
> Status: Implementation ready

## 1. What is the Cortex?

A middleware cognitive layer WITHIN each NEXO session that makes the agent think before acting.
Not a separate process, not a daemon, not a background service.

It is the difference between "agent with memory" and "agent that reasons about when and how to act."

## 2. Core Principle

**The Cortex does NOT generate answers. It gates, plans, and validates actions.**

```
User message arrives
    ↓
Fast Path check: is this a simple chat response?
    ├─ YES → respond normally (zero overhead)
    └─ NO → Cortex activates
              ↓
         Generate cognitive state (goal, plan, unknowns, constraints)
              ↓
         Middleware evaluates state
              ├─ missing_slots? → force clarification (block tools)
              ├─ destructive + no verification? → propose only (block execute)
              ├─ plan ready + evidence present → allow execution
              └─ contradiction detected → force re-evaluation
              ↓
         Agent acts (with tools enabled/disabled per state)
              ↓
         Post-action check: goal met? contradictions? unsupported claims?
              ↓
         Deliver response
```

## 3. When Does Cortex Activate? (Event-Driven)

Cortex is NOT invoked on every turn. It activates ONLY when:

1. **Tool intent detected** — agent is about to edit files, run commands, call APIs
2. **Ambiguity detected** — user request has multiple interpretations
3. **Destructive action** — file deletion, git push, DB mutation, deploy
4. **Multi-step task** — user asks for something requiring >3 tool calls
5. **Retry after failure** — previous attempt failed, need different approach
6. **Contradiction** — user instruction conflicts with known facts/learnings

Simple chat, greetings, questions, explanations → NO Cortex overhead.

## 4. Cognitive State Object

Generated by the agent before action, validated by middleware:

```json
{
  "goal": "what am I trying to achieve",
  "task_type": "answer|analyze|edit|execute|delegate",
  "plan": ["step 1", "step 2", "step 3"],
  "known_facts": ["fact from user", "fact from memory"],
  "unknowns": ["what I don't know yet"],
  "constraints": ["rule or limitation that applies"],
  "evidence_refs": ["learning #X", "user said Y", "file Z shows W"],
  "verification_step": "how to verify this worked",
  "next_action": "ask|propose|act"
}
```

### Field validation rules (middleware enforces):
- `plan` must have ≥1 step if `task_type` is edit/execute/delegate
- `unknowns` non-empty → `next_action` forced to "ask" (tools blocked)
- `task_type` is "edit" and `verification_step` empty → `next_action` forced to "propose"
- `evidence_refs` empty and `task_type` is "execute" → `next_action` forced to "propose"
- `constraints` should include relevant Core Rules (auto-injected by middleware)

## 5. Inhibitory Control (Architectural, Not Instructional)

The agent cannot "choose" to ignore inhibition. The middleware physically controls tool availability:

| Condition | Tools Available | Agent Can |
|-----------|----------------|-----------|
| `next_action = "ask"` | Read, Search only | Ask questions, gather info |
| `next_action = "propose"` | Read, Search, Analyze | Propose plan, show diff, explain |
| `next_action = "act"` | All tools | Execute the plan |

### Escalation from ask → propose → act:
1. Agent generates state with unknowns → middleware sets "ask" mode
2. User provides clarification → agent regenerates state without unknowns → middleware allows "propose"
3. User approves plan (or plan is low-risk) → middleware allows "act"

### Override: User explicitly says "just do it" → bypass to "act" mode regardless of state.

## 6. Persistence

- **Per-turn**: full cognitive state lives in context only (no DB write)
- **Cross-turn carryover**: if task spans multiple turns, persist ONLY:
  ```json
  {
    "active_task": "goal description",
    "current_step": 2,
    "open_questions": ["what user hasn't answered yet"],
    "constraints": ["accumulated constraints"]
  }
  ```
  Stored as temporary record in SQLite, TTL 1 hour, deleted on task completion.
- **NEVER** persist cognitive states to LTM/STM (that's for learnings and memories, not transient cognition)

## 7. Interaction with Existing Systems

| System | Relationship |
|--------|-------------|
| **30 Core Rules** | Cortex enforces them contextually. Rules are law; Cortex is judge. |
| **Guard System** | Guard checks learnings before code edits. Cortex decides IF editing is appropriate first. Guard runs AFTER Cortex approves "act" mode. |
| **Trust Score** | Low trust → Cortex requires more evidence and stricter verification. High trust → Cortex allows faster escalation to "act". |
| **Context Packet** | Cortex uses context_packet output to populate `known_facts` and `constraints`. |
| **Smart Startup** | Feeds initial context. Cortex doesn't run at startup — only on action triggers. |

## 8. Implementation Plan

### Phase 1: MCP Tool (`nexo_cortex_check`) — v0.1
A single MCP tool that the CLAUDE.md instructs the agent to call before significant actions.

```
nexo_cortex_check(cognitive_state: JSON) → result with allowed tools and warnings
```

The tool:
1. Validates the cognitive state JSON
2. Applies inhibition rules (missing slots, no verification, no evidence)
3. Returns: `mode` (ask/propose/act), `warnings` (what's missing), `injected_rules` (relevant Core Rules)

### Phase 2: Automatic Activation — v0.2
Hook into PostToolUse/PreToolUse to detect action intent and auto-trigger Cortex evaluation without explicit tool call.

### Phase 3: Context Masking — v0.3
Selective attention: trim irrelevant memories from context based on current goal. Reduces token waste.

## 9. Day-1 Value (New User)

Without history or learnings:
- Cortex forces the agent to **scope before acting** — "what do I know? what don't I know?"
- Empty `known_facts` = agent must ask questions instead of guessing
- `verification_step` required = agent plans how to check its work
- Core Rules injected = behavioral discipline from the first interaction

## 10. Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Overhead on simple tasks | Event-driven activation; simple chat bypasses entirely |
| LLM ignores state object | CLAUDE.md makes it mandatory; middleware validates before allowing tools |
| Self-reported confidence is unreliable | No confidence field — use observable predicates (missing_slots, evidence_refs) |
| State pollution of memory | Cognitive states NEVER go to LTM/STM; TTL 1h for cross-turn persistence |
| Over-engineering | v0.1 is ONE tool + middleware validation. No multi-LLM, no neural pathways. |

## 11. Success Metrics

After 2 weeks of use:
- Reduction in "premature action" errors (baseline from current trust score events)
- Reduction in repeated learnings (agent checks before acting)
- User satisfaction (trust score trend)
- Cortex activation rate (should be 20-40% of turns, not 100%)
- False inhibition rate (Cortex blocking when it shouldn't — target <5%)
