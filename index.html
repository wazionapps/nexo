Why Teams Keep NEXO Running

Not a plugin. A local operating layer for your agent.

The hard part is not storing facts. The hard part is staying coherent after compaction, sleep, updates, and autonomous background work.

Persistent memory that decays naturally

NEXO keeps learnings, followups, reminders, entities, and episodic history locally, with semantic retrieval, reinforcement, and forgetting instead of infinite clutter.

Startup preflight and runtime doctor

Before interactive use, NEXO can run safe local preflight steps. When something drifts, nexo doctor explains it and can repair deterministic issues.

Recovery-aware automation

Core jobs and personal schedules declare explicit recovery contracts for boot, wake, catch-up, restart, idempotency, and missed-run windows.

Personal scripts as first-class citizens

Your scripts in NEXO_HOME/scripts/ are managed, scheduled, reconciled, diagnosed, and recovered through the official flow instead of ad-hoc plist drift.

What Changed Recently

2.6.5 closes the gaps users actually feel

The latest release line is not about vanity features. It is about safer runtime behavior when the host sleeps, wakes, updates, and recovers.

Startup preflight before chat and server

NEXO now runs a shared startup path that can apply safe local migrations, defer risky remote updates, and keep the runtime sane before interactive work begins.

Catch-up recovery for core and personal schedules

Recovery contracts now cover both core jobs and managed personal schedules, with wake-aware policies like run_once_on_wake and catchup.

Power helper semantics are honest now

always_on means best-effort background availability through the platform helper. It helps, but NEXO still treats wake recovery as mandatory.

No more duplicate catch-up relaunches

Catch-up now detects in-flight cron windows and suppresses duplicate recovery launches, which closes the double-run bug observed in real use.

FAQ

Questions people ask before they install

Short answers, current product surface, no outdated claims.

Is NEXO only memory?
No. NEXO is a local cognitive runtime. Memory is one part of it. The rest is runtime operations, startup preflight, doctor diagnostics, overnight learning, recovery-aware jobs, and personal automation.
Does NEXO send my data to the cloud?
NEXO stores its own state locally using SQLite and local embeddings. It is built as a local-first memory and operations layer.
What happens if my laptop sleeps through a cron window?
Eligible jobs can be recovered on boot or wake through catch-up. Managed personal scripts can also declare explicit recovery behavior instead of relying on naive interval scheduling.
Can I automate my own scripts?
Yes. Scripts in NEXO_HOME/scripts/ are first-class managed objects with metadata, schedules, doctor checks, reconcile, and wake-aware recovery policies.
Does always_on guarantee closed-lid operation on a laptop?
No. It enables the platform power helper for best-effort background availability. On macOS that means caffeinate. Wake recovery and catch-up still remain part of the operational contract.
How do updates work?
Use nexo update explicitly. NEXO also has startup preflight logic that can apply safe local migrations and defer risky remote update work when the runtime is busy.
Is the benchmark real?
Yes. NEXO reached F1 0.588 on LoCoMo, outperforming GPT-4 128K in the published comparison while running on CPU only. See the benchmark page for context and links.
What is the fastest way to evaluate it?
Install with npx nexo-brain, run nexo chat, then inspect nexo doctor --tier runtime --json. That shows both the memory surface and the operational runtime.
